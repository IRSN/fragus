# Evaluation data

This directory holds the input dataset and the output files of the RAG evaluation
campaign. It is intentionally (almost) empty in the public repository: the original
ASNR evaluation dataset and the raw results derive from the ingested corpus and are
**not published** (see the note at the bottom).

## 1. Input dataset: `data/rag_evaluation_dataset.xlsx`

The question bank consumed by `prepare_eval_dataset.py` (step 1 of the campaign).
It is a single-sheet Excel file (the first sheet is read by default; use `--sheet`
to select another one).

Required columns:

| Column | Description |
|---|---|
| `q_id` | Unique question identifier (string), e.g. `Q001`. Also used as the default `paraphrase_group_id` and as the resume key in the output file. |
| `user_input` | The question text, as an end user would ask it. |
| `reference` | The expected answer (ground truth), written by a domain expert. Used by the judge LLM for factual correctness, context recall/precision and anchoring. |
| `reference_files` | The source document names expected to support the answer. Either a JSON list (`["report_A.pdf", "technical_guide_B.pdf"]`) or one file name per line inside the cell. Used for the file-level citation and context metrics. |

Optional columns:

| Column | Description |
|---|---|
| `test_intent` | Free-form intent category of the question (e.g. factual lookup, synthesis, out-of-scope). Carried through to the score files for per-intent analysis. |
| `paraphrase_group_id` | Shared identifier linking a question and its reformulations. Defaults to `q_id` when absent. |
| `is_paraphrase` | `False` (or empty) for original questions, `True` for hand-written reformulation rows. Accepts booleans, `0`/`1` or `"true"`/`"false"`. Pre-existing variant rows are reused as-is; if a group has fewer variants than `--paraphrase-variants` (default 3), the missing ones are generated by an LLM. |

To build your own evaluation dataset, provide the four required columns for each
original question; paraphrase variants are optional (they enable the
`paraphrase_robustness` metric and can be generated automatically).

## 2. Output files (`data/results/`)

The two-step campaign produces, for a given `--experiment-id <id>`:

### `eval_test_<id>.xlsx` — produced by `prepare_eval_dataset.py`

One row per question *and* per paraphrase variant, with the RAG pipeline's output.
Columns (canonical RAGAS names, directly consumable by `rag_evaluation.py`):

- `q_id`, `test_intent`, `paraphrase_group_id`, `is_paraphrase`, `user_input`,
  `reference`, `reference_files` — carried over from the input (variants get ids
  like `Q001_v1`);
- `response` — the answer generated by the RAG pipeline;
- `retrieved_contexts` — JSON list of the chunk texts passed to the LLM (after reranking);
- `retrieved_files` — JSON list of the unique document names present in the retrieved chunks;
- `cited_files` — JSON list of the document names cited via `[N]` markers in the response.

The file is saved incrementally and doubles as a resume checkpoint: re-running the
script with the same experiment id skips already-answered questions.

### `score_test_<id>_per_question.xlsx` — produced by `rag_evaluation.py`

One row per **original** question (variants only feed `paraphrase_robustness`),
with identification columns (`q_id`, `test_intent`, `user_input`) and one column
per metric:

- RAGAS: `faithfulness`, `context_recall_llm`, `context_precision_llm` (+ derived
  `hub_france_ia_fscore`);
- factual correctness (recomputed internally): `factual_correctness_precision`,
  `factual_correctness_recall`, `factual_correctness_f1`;
- file-level context: `context_relevance_files`, `context_coverage_files`;
- file-level citations: `citation_precision_files`, `citation_recall_files`,
  `citation_fscore_files`;
- embeddings: `diversity`, `paraphrase_robustness`;
- anchoring (LLM fact classification): `anchor_expected_grounded`,
  `anchor_expected_ungrounded`, `anchor_additional_grounded`,
  `anchor_additional_ungrounded`, `hallucination_rate`, `grounded_rate`;
- `global_rag_score` — the 4-component weighted aggregate.

This file also serves as the checkpoint of the scoring run: rows already scored
are not recomputed when the evaluation is re-launched with the same prefix.

### `score_test_<id>_averages.xlsx` — produced by `rag_evaluation.py`

One row per metric: `metric` | `mean` (NaN ignored) | `n_valid` (number of
questions with a valid value).

### Other files

- `comparison_<A>_vs_<B>.xlsx` — paired statistical comparison of two runs
  (Wilcoxon + Benjamini-Hochberg), produced by `compare_rag_runs.py`.
- `query_embeddings_cache.pkl`, `emb_cache_*.pkl` — local embedding caches used
  to speed up repeated runs; safe to delete, never versioned.

## 3. Note on the original ASNR data

The original ASNR evaluation dataset (`rag_evaluation_dataset.xlsx`) and the raw
result files (`eval_test_*.xlsx`, `score_test_*.xlsx`, `*.pkl`) contain content
derived from the ingested document corpus and are therefore not published in this
repository. The aggregate scores of all evaluation runs are available in
[docs/rag_runs_catalog.md](../../../docs/rag_runs_catalog.md).
